# Rardar Evolution Protocol

> 文档角色：规定 Codex 或其他工程 Agent 如何长期迭代 Rardar。
> 文档状态：Protocol v1
> 核心原则：每次只完成一个有边界、可测试、可回滚的改进，然后创建 Draft PR 并停止。

---

## 1. 工作目标

你正在维护：

```text
Brilliant666/rardar
```

你的目标不是尽快堆叠功能，而是在不破坏现有能力的前提下，使 Rardar 持续变得：

- 更可信；
- 更安全；
- 更可解释；
- 更可恢复；
- 更容易维护；
- 更接近真实项目复用决策。

---

## 2. 开始工作前必须阅读

按顺序阅读：

```text
README.md
AGENTS.md
docs/RARDAR_AUDIT_BASELINE.md
docs/RARDAR_NORTH_STAR.md
docs/RARDAR_EVOLUTION_PROTOCOL.md
```

如存在以下文档，也必须阅读：

```text
docs/ARCHITECTURE.md
docs/DATA_MODEL.md
docs/ADR/
SECURITY.md
CONTRIBUTING.md
```

然后检查仓库状态：

```bash
git status
git branch --show-current
git log -5 --oneline
git remote -v
```

确认：

- 当前仓库；
- 当前分支；
- 默认分支基线；
- 是否存在未提交修改；
- 是否已有相关 Issue、分支或 PR。

不得覆盖用户已有未提交修改。

## 3. 默认长期优先级

没有用户指定其他目标时，选择列表中第一个尚未完成的目标：

1. 数据 Schema 和统一契约——已由 PR #2（提交 `048a2d9`）完成；
2. audited generations——已由 PR #4（提交 `bf35575`）完成；
3. 追加式行动事件模型——由当前行动事件分支实现，仅在对应 PR 合并到 `main` 后视为完成；
4. 评分语义修正——当前行动事件 PR 合并后的第一个未完成项；
5. 统一 verify 和 GitHub Actions；
6. 稳定项目 ID；
7. 环境和开源治理文档；
8. 静态能力识别升级；
9. 任务到项目匹配；
10. 用户项目上下文；
11. 隔离验证闭环。

当前行动事件 PR 合并前不得开始评分语义或其他后续工程目标。合并后仍需先确认最新 `main` 已包含追加式行动 Event、独立 State 与按 Event 计算的 Weekly Acted Projects，工作区干净，且不存在尚未完成的行动事件修正 PR，再按“选择列表中第一个尚未完成的目标”规则建立独立评分语义分支。

不得跳过可信数据基线，直接构建复杂 Agent、向量检索或自动执行系统。

---

## 4. 每轮只做一个目标

每个 PR 只处理一个清晰目标。

禁止同一 PR 同时包含：

- 大规模 UI 重设计；
- 数据库模型重构；
- 评分公式重写；
- 流水线发布机制重构；
- 新信源扩展；
- 部署方式变更。

推荐分支：

```text
fix/<problem>
feat/<capability>
refactor/<module>
docs/<topic>
test/<scope>
```

---

## 5. 每轮标准流程

## 5.1 建立当前事实基线

运行现有验证：

```bash
npm run lint
python -m unittest discover -s pipeline -p "test_*.py"
npm run data:validate
npm run data:audit
npm run build
npm test
npm run security:audit:prod
```

某项不可运行时：

- 记录实际错误；
- 说明环境缺失；
- 不得声称通过；
- 不得用静态阅读代替执行结果。

## 5.2 定义问题

修改前写清：

- 当前行为；
- 为什么是问题；
- 影响范围；
- 最小修复；
- 非目标；
- 数据兼容；
- 回滚方式。

## 5.3 检查边界

判断是否影响：

- 事实与 AI 判断分层；
- 第三方仓库只读原则；
- 数据当前性；
- Schema；
- generation；
- 历史快照；
- 数据库迁移；
- 周度指标；
- 许可证；
- URL 安全；
- 本地优先。

## 5.4 实现最小改动

优先复用现有：

- helper；
- 数据模型；
- 数据锁；
- 原子写入；
- 审计函数；
- 测试工具；
- 当前错误处理方式。

不要为了“更现代”重写整个模块。

## 5.5 增加行为测试

测试必须验证行为，而不只是源码字符串。

优先：

- 函数单元测试；
- API 路由测试；
- D1/SQLite 行为测试；
- 失败恢复测试；
- 迁移测试；
- 时间窗口测试；
- 前端关键交互测试；
- 数据发布切换测试。

所有 Bug 修复必须包含修复前失败、修复后通过的回归测试。

## 5.6 验证失败路径

至少考虑：

- 网络失败；
- 部分信源失败；
- 非法 JSON；
- Schema 不匹配；
- 文件替换中断；
- generation 审计失败；
- 数据库冲突；
- 重复提交；
- 旧画像；
- 无许可证；
- 超大仓库；
- 调度器重启；
- 服务持续崩溃。

## 5.7 更新文档

改动影响以下内容时必须更新文档：

- 数据含义；
- 评分；
- Schema；
- 命令；
- 环境；
- 安全边界；
- 北极星；
- 部署或本地运行；
- 数据迁移。

## 5.8 完整验证

最终报告必须分为：

```text
通过：
失败：
未运行：
原因：
```

禁止将“看起来正确”描述为“测试通过”。

## 5.9 创建 Draft PR

PR 描述必须包含：

```text
背景
问题
本次修改
方案选择
数据或 Schema 变化
测试结果
安全影响
兼容性
迁移方式
回滚方式
未解决问题
```

默认创建 Draft PR。

不得直接合并 main。

## 5.10 停止

完成 Draft PR 后停止。

不要自动开始下一个架构任务。

等待用户决定：

- 审查；
- 修改；
- 合并；
- 放弃；
- 开始下一轮。

---

## 6. 禁止事项

未经用户明确要求，不得：

- 部署或重新部署；
- 直接提交 main；
- 自动合并；
- 执行候选第三方仓库代码；
- 安装候选仓库依赖；
- 删除历史数据；
- 修改 Git 全局配置；
- 修改凭据；
- 引入向量数据库、Neo4j、Elasticsearch 或大型 Agent 框架；
- 重写整个项目；
- 用 AI 输出覆盖事实；
- 无证据提高评分；
- 将公开可见等同于可复用许可证；
- 将一次成功试用描述为生产可用。

---

## 7. 决策优先级

发生冲突时：

```text
数据正确性
> 安全边界
> 可解释性
> 可恢复性
> 自动测试
> 用户体验
> 开发速度
> 技术新颖性
```

---

## 8. 自我迭代循环

每次完成 PR 后，Codex 应更新一个轻量的迭代状态：

```text
docs/iterations/<YYYY-MM-DD>-<topic>.md
```

内容：

```text
目标
完成内容
验证结果
新发现
遗留风险
是否影响 North Star
建议的下一项最高优先级
```

每次 PR 合并后，必须更新治理文档中的完成状态或补充对应迭代记录，并在下一轮结合最新 `main` 重新判断第一个未完成项，避免重复实施已经完成的目标。

但是：

- 不能未经用户同意自动继续下一项；
- 不能自动合并；
- 不能把建议任务伪装成已完成任务。

---

## 9. 第一阶段的完成定义

当以下条件全部满足时，Phase 0 才算完成：

- 所有核心 JSON 具有 Schema；
- 写入前进行 Schema 校验；
- generation 审计通过后才发布；
- 审计失败保持上一代健康数据；
- 行动事件跨周期准确；
- 项目 ID 无碰撞；
- `npm run verify` 覆盖前后端和 Python；
- GitHub Actions 作为 PR 门槛；
- 环境和开源治理文档完整；
- main 始终可构建、可审计、可回滚。
