# 2026-07-14 · Repeatable Verify and CI gate

## 目标

建立单一、可重复且可审计的仓库级验证入口，并让指向 `main` 的 Pull Request 与 push 到 `main` 通过同一 GitHub Actions 门禁验证；验证不得依赖或修改 Primary Runtime、正式数据、凭据或实时业务刷新。

## 本 PR 交付

- 新增唯一推荐完整入口 `npm run verify`；
- 顺序复用 lint、Python 测试、Schema、Audit、production build、Node 测试和 production dependency security audit，任一失败返回非零；
- 将不含 build 的 Node 套件抽为 `test:node`，Verify 只构建一次，原有 `npm test` 行为保持为 build 后运行同一 Node 套件；
- 使用无 shell 的 Node 子进程编排，Windows PowerShell 与 Linux runner 共用同一实现；
- 在 Verify 前后递归比较整个 `data/` 的路径、类型和 SHA-256，并比较 Git porcelain 状态；
- 将 `LOCALAPPDATA`、`HOME`、XDG、Rardar Runtime、Wrangler、Miniflare、Vinext 与临时状态隔离到一次性目录，移除传给子进程的 GitHub/npm token，并在 `finally` 中清理；
- 新增完整固定直接与传递版本的 `requirements.lock`，并由最小权限 GitHub Actions 使用 Node/Python 锁文件安装，最终只调用 `npm run verify`；
- 本地 Verify 通过绝对 `RARDAR_PYTHON` 使用 worktree 自有 `.venv`；CI 以最低支持版本 Python 3.10 创建一次性 `.venv-ci`，不依赖系统级 Python 包或 Primary Runtime 虚拟环境；
- 更新直接相关的 README、治理状态和本迭代记录，不改变产品、评分、Catalog、generation、Event 或 Stable ID 语义。

## 门禁与 CI

GitHub Actions 在以下事件运行单一 Ubuntu job：

- Pull Request 指向 `main`；
- push 到 `main`。

workflow 使用 `contents: read`、30 分钟超时、同一 PR/ref 新运行取消旧运行、固定提交的官方 checkout/setup actions、Node.js 22.13.1、Python 3.10 和 `npm ci`。它显式创建一次性 `.venv-ci`，使用 `.venv-ci/bin/python` 安装 `requirements.lock` 并执行 `pip check`，再通过绝对 `RARDAR_PYTHON` 运行唯一一次 `npm run verify`。缓存不替代安装；workflow 不依赖系统级 Python 包，也不启动长期服务，不调用 manager、scheduler、refresh、derive、generation publish/rollback、3000 端口或部署。

- Audit：generation `20260713T191537075729Z-4e2e9d09fae2` 为 `healthy`，0 error、0 warning；
- Production build：通过；
- Node：17/17 通过，真实 Vinext HTTP 验收使用随机回环端口与临时 data/D1；
- Production dependency security audit：0 个已知生产依赖漏洞；
- 数据保护：整个 `data/` 前后完全一致；
- 工作区保护：Verify 未留下新的 Git 可见产物；
- Runtime 保护：一次性隔离状态成功删除。
- 失败传播：使用不存在的 Python 可执行文件进行负向检查时，Verify 返回 1，仍完成数据、Git 与 Runtime 清理保护。

Windows 跳过项逐项为：

这些测试没有按平台永久排除；Ubuntu GitHub Actions 会实际创建并验证对应链接路径。workflow 已通过本地 YAML 结构解析和静态字段检查；远端 Actions 结果以 Draft PR 的真实 checks 为准，本地记录不提前声称远端通过。

## 基线与新发现

- 初始 worktree 的 `node_modules` 缺少 lockfile 声明的 Windows Rolldown 可选原生包，首次 build 与 Node 门禁因此失败；按 `package-lock.json` 执行 `npm ci` 后，未修改 lockfile，build 与 17 项 Node 测试均通过；
- Python 运行时范围声明只有 `jsonschema>=4.17,<5`，干净 Windows Verify 环境还缺少 `ZoneInfo("Asia/Shanghai")` 所需的系统时区数据库；本 PR 不改变 manager/Runtime 预检，而是在经官方 PyPI 与隔离 venv 验证的 Verify/CI `requirements.lock` 中固定 `tzdata`，共锁定 7 个直接与传递包；
- 首次以绝对 venv Python 路径运行时暴露了 Windows 环境键 `Path` 的大小写差异；编排器现按大小写无关方式更新既有 PATH，并让 Python tests、Schema、Audit 与 Node HTTP 子进程统一使用该锁定解释器；
- production security audit 按现有政策硬失败，只检查会进入运行环境的依赖，不使用 `continue-on-error`；
- workflow check 只有被 `main` 分支保护设为 required 后才真正阻止失败合并，远端保护状态必须与 PR checks 一起核对。
