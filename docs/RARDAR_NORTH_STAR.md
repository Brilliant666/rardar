# Rardar North Star

> 文档角色：定义 Rardar 的长期使命、不可退让原则、核心指标和阶段路线。
> 文档状态：North Star v1
> 本文优先级高于临时功能需求，但低于用户在当前任务中的明确指令。

---

## 1. 项目使命

Rardar 是一个证据优先、面向个人开发者和小型工程团队的开源软件情报与项目复用决策系统。

Rardar 不只是回答：

- 最近有哪些热门项目？
- 哪些项目 Star 增长最快？

Rardar 最终需要回答：

- 最近真正发生了什么？
- 某项能力是否已有成熟实现？
- 哪个项目最适合当前任务？
- 哪些模块可以独立复用？
- 复用成本、许可证、安全和维护风险是什么？
- 应该直接采用、隔离试用、浅克隆研究，还是继续观望？
- 该项目如何接入用户已有项目？

---

## 2. 长期产品定位

> 个人开发者的开源软件情报、能力发现和复用决策中枢。

核心闭环：

```text
公开技术生态
→ 事实采集
→ 静态证据
→ 可解释排序
→ Codex 深度分析
→ 能力画像
→ 任务匹配
→ 用户实际行动
→ 验证结果和复用结果反馈
```

Rardar 的价值不以页面浏览量衡量，而以是否帮助用户采取有效工程行动衡量。

---

## 3. 北极星指标

## 3.1 主指标

### Weekly Acted Projects

近 7 天发生以下任一行动的不同项目数：

- 实际试用；
- 只读浅克隆或静态检查；
- 确认复用到自己的任务或项目。

中文名称：

> 近 7 天已行动项目数

### 指标要求

- 必须来自追加式历史事件；
- 不能由当前状态倒推；
- 同一个项目在不同周再次行动，应在对应周期重新计入；
- 网络重试不得造成重复计数；
- 反馈“有用/无用”不能冒充工程行动。

## 3.2 辅助指标

1. Daily Five 中当前有效静态证据覆盖率。
2. Daily Five 中当前有效中文能力画像覆盖率。
3. 每日数据刷新成功率。
4. 发布 generation 审计通过率。
5. 打开 → 试用 → 浅克隆 → 确认复用转化率。
6. 因仓库更新而自动降级或重新入队的项目数。
7. 推荐后被标记无用的比例。
8. 已确认复用项目的持续使用率。
9. 数据源健康率。
10. 静态分析失败率和超限率。

## 4. 不可退让的产品原则

## 4.1 事实与判断分离

事实层包括：

- GitHub API 数据；
- RSS 和官方发布时间；
- Star、Fork、Issue；
- 仓库创建、更新和推送时间；
- 许可证 API 结果；
- 文件结构；
- 静态依赖和框架证据。

判断层包括：

- 中文摘要；
- 能力画像；
- 推荐理由；
- 复用计划；
- 限制；
- 风险判断；
- 任务适配度。

AI 判断不得覆盖、改写或伪造事实层。

## 4.2 证据先于结论

没有当前证据时：

- 不显示“深度分析”；
- 不给出确定性能力结论；
- 不建议直接复用；
- 必须明确标记事实初筛、静态分析不足或画像待复核。

## 4.3 陌生代码默认不执行

第三方候选仓库默认：

- 不安装依赖；
- 不运行脚本；
- 不运行测试；
- 不执行构建；
- 不加载 Hook；
- 不使用用户 Git 全局配置；
- 不读取用户凭据；
- 只进行有资源边界的静态检查。

实际运行必须进入独立验证流程或沙箱，不得由默认雷达流水线执行。

## 4.4 所有评分必须可解释

每个评分必须能够回答：

- 使用了哪些事实？
- 使用了哪些代理指标？
- 哪些信息未经验证？
- 为什么得到当前分数？
- 哪些因素限制了上限？
- 哪些条件满足后分数才可升级？

禁止只输出不可解释的总分。

## 4.5 审计通过后才发布

新一代数据必须遵守：

```text
生成
→ Schema 校验
→ 一致性审计
→ 全部通过
→ 原子发布
```

正式发布以不可变 generation 和单一 `data/current.json` 指针为边界。网页一次请求与下一轮增长计算都必须从一次指针解析得到同一 generation；不得分别回退到 flat 文件。并发发布需要比较候选的 base generation，回滚只能显式指向重新验证通过的保留代。

审计失败的数据：

- 不得替换上一代健康数据；
- 不得被网页读取；
- 不得成为增长基线；
- 不得归档上一代；
- 必须保留可诊断状态。

## 4.6 事件与状态分离

- Event 表记录发生过什么。
- State 表记录现在是什么状态。
- 指标基于事件。
- UI 状态基于 State。
- 不得用唯一状态表代替时间序列事件。

## 4.7 个性化不能覆盖事实主干

个性化可以有限重排，但不得：

- 隐藏风险；
- 绕过许可证限制；
- 将过期画像恢复为当前画像；
- 让无证据项目压过高证据项目；
- 将用户偏好描述成全球趋势。

## 4.8 本地优先和隐私优先

未经用户明确要求：

- 不自动部署；
- 不修改生产环境；
- 不上传本地行为；
- 不上传本地 Codex 画像；
- 不接入外部遥测；
- 不公开用户项目上下文。

## 5. 核心领域模型

Rardar 应逐步形成：

```text
RepositoryIdentity
GitHubFactSnapshot
TechnicalSignal
StaticEvidence
LicenseEvidence
CapabilityEvidence
CodexEnrichment
TaskIntent
ScoreExplanation
UserFeedbackState
UserActionEvent
ReuseDecision
ValidationResult
DataGeneration
```

每个对象至少包含：

- `schemaVersion`
- 稳定标识
- 来源
- 创建时间
- 当前性字段
- 过期判断
- 证据链接
- 可审计字段定义

## 6. 评分体系目标

## 6.1 Attention Score

衡量“现在是否值得关注”。

依据可包括：

- 实际观察区间增长；
- 新鲜度；
- 最近维护；
- 多入口召回；
- 官方技术事件关联；
- 风险降权。

## 6.2 Endurance Score

衡量“是否具有长期生态价值”。

依据可包括：

- 仓库年龄；
- 总 Star；
- Fork 生态；
- 维护持续性；
- 多周期候选覆盖率；
- Release 活跃度。

## 6.3 Engineering Readiness

衡量“静态工程证据是否完整”。

依据可包括：

- README；
- 许可证；
- 测试；
- CI；
- 文档；
- 示例；
- 包清单；
- 依赖锁；
- 发布流程；
- 安全策略。

它不能被描述为实际运行可靠性。

## 6.4 Reuse Fit Score

衡量“对具体任务或已有项目的复用适配度”。

依据应包括：

- 功能匹配；
- 技术栈匹配；
- 模块可拆分性；
- API、SDK、CLI 边界；
- 依赖复杂度；
- 许可证；
- 安全边界；
- 接入成本；
- 实际验证结果。

没有具体任务上下文时，不应给出过高的 Reuse Fit Score。

## 7. 阶段路线

## Phase 0：可信数据基线

### 目标

保证每一代正式数据都可验证、可回滚、可重复构建。

### 必须完成

1. 所有 JSON 产物建立正式 Schema。
2. Codex 输出严格类型验证。
3. generation 生成、审计、原子发布。
4. 稳定且抗碰撞的项目 ID。
5. 行为事件改为追加式。
6. 建立统一 `npm run verify`。
7. 建立 GitHub Actions。
8. 补齐环境、LICENSE、SECURITY、ARCHITECTURE、DATA_MODEL。

### 完成标准

- 审计失败不会污染正式数据；
- 所有验证由一个命令运行；
- PR 自动运行验证；
- 新环境可按文档启动；
- 周度行动指标跨周期准确。

## Phase 1：能力识别

### 目标

从仓库工程结构识别真实能力，不只统计文件存在。

### 增加

- 依赖和框架检测；
- API、CLI、SDK 入口；
- 数据库和外部服务；
- 模块结构；
- 部署方式；
- 许可证作用范围；
- README 声明与代码证据差异；
- 可拆分模块候选。

### 完成标准

重点项目具有结构化、可追踪的能力证据。

## Phase 2：任务到项目匹配

### 目标

将自然语言目标拆成能力需求，并匹配项目或项目组合。

### 增加

- 任务意图；
- 必需能力；
- 可选能力；
- 单仓库方案；
- 多仓库组合；
- 接入成本；
- 风险；
- 缺失能力。

### 完成标准

每个推荐能够解释“为什么匹配”和“还缺什么”。

## Phase 3：个人项目上下文

### 目标

理解用户已有项目、技术栈和历史选择。

### 增加

- 用户项目画像；
- 当前依赖；
- 模块边界；
- 部署限制；
- 历史试用和复用结果；
- 与候选项目的兼容性。

### 完成标准

能够回答某项目哪些模块可接入用户现有项目。

## Phase 4：验证闭环

### 目标

把发现、推荐和实际验证连接起来。

```text
Rardar 发现
→ 生成验证任务
→ platform_lab / 沙箱验证
→ 保存 ValidationResult
→ 更新复用建议和评分
```

### 完成标准

页面明确区分：

- 未验证；
- 静态证据；
- 隔离试用；
- 已验证；
- 已确认复用。

---

## 8. 明确非目标

现阶段不做：

- 默认自动执行第三方仓库代码；
- 自动安装陌生依赖；
- 无边界全网爬虫；
- 用 LLM 摘要替代事实采集；
- 为了数量盲目扩充信源；
- 没有任务上下文却声称直接复用；
- 提前引入复杂分布式基础设施；
- 同一 PR 同时重写前端、流水线和数据库；
- 未经要求部署线上版本；
- 将公开可见误认为拥有复用许可证。

---

## 9. 架构演进门槛

## 保持 JSON 的条件

- 本地重建速度可接受；
- 写入是每日批次；
- 不需要复杂并发查询；
- 历史体积可控制；
- 审计可在合理时间完成。

## 迁移 SQLite 的条件

- 历史快照明显膨胀；
- 时间序列查询频繁；
- 能力、事件和项目需要关联查询；
- JSON 全量重建成为瓶颈。

## 迁移 PostgreSQL 的条件

- 出现多用户；
- 出现远程服务；
- 出现并发写入；
- 出现团队共享、认证和权限。

不得因为技术流行而提前迁移。

---

## 10. 每轮迭代的维护要求

每次有效工程迭代必须检查：

1. 实现代码；
2. 自动测试；
3. 数据 Schema；
4. 数据审计；
5. 迁移与兼容；
6. 文档；
7. 安全边界；
8. 回滚方式；
9. 可观察状态；
10. 遗留问题。

如果改动影响：

- 北极星；
- 评分；
- 数据含义；
- 安全边界；
- 发布流程；

必须同步更新本文。
